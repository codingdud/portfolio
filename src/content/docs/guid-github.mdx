# Mentioning Issues in Commit Messages & Contributing to Projects on GitHub

Contributing to a project on GitHub often involves making changes that relate to specific issues or bugs. Properly mentioning these issues in your commit messages or pull requests helps maintain clarity and traceability for the entire team.

This guide explains how to reference issues in commit messages and during the contribution process.

---

## ğŸ“Œ Why Mention Issues?

When you mention an issue in a commit message or pull request, you:

- **Link your work** directly to the problem it solves.
- Help maintainers understand **why** a change was made.
- Automatically update issue threads with links to your commits or PRs.
- Enable **automatic closure** of issues when your code is merged.

---

## ğŸ”— How to Reference an Issue in a Commit Message

GitHub recognizes certain keywords followed by an issue number in commit messages. These can be used to automatically link or close issues.

### âœ… Supported Keywords:
- `close`
- `closes`
- `closed`
- `fix`
- `fixes`
- `fixed`
- `resolve`
- `resolves`
- `resolved`

### ğŸ§¾ Syntax:
Use one of the keywords followed by `#<issue-number>`.  
You can also reference issues from other repositories using `username/repo#<issue-number>`.

### ğŸ’¡ Example Commit Messages:
```bash
git commit -m "Fix bug in login flow. Closes #45"
```

```bash
git commit -m "Update dependencies. Fixes tonychacon/blink#12"
```

If this commit is pushed to a branch and later merged into the default branch (e.g., `main` or `master`), **the referenced issue will be automatically closed**.

---

## ğŸ”„ Referencing Issues Without Closing Them

If you want to reference an issue without closing it, just use the `#<number>` syntax without any keyword.

### ğŸ’¬ Example:
```bash

```

This will create a link in the issue thread but wonâ€™t close the issue.

## ğŸ“¥ Using Issue References in Pull Requests

When creating or updating a Pull Request:

- In the **title or description**, include `Closes #<issue-number>` to indicate that merging the PR will resolve the issue.
- You can also mention multiple issues:
  ```md
  This fixes #12 and addresses #45.
  ```

> ğŸ’¡ Tip: When you merge a PR that says `Closes #12`, GitHub will automatically close Issue #12 and link back to the PR from the issue.

---

## ğŸ¯ Best Practices

- Always write **clear, descriptive commit messages** even if you reference an issue.
- Use `Closes #<issue>` only when your change fully resolves the issue.
- If you're working across forks or repositories, use full references like `username/repo#12`.
- You can reference **multiple issues** in a single commit or PR:
  ```bash
  git commit -m "Fix performance issues. Closes #10, #12"
  ```
